Todos os Arquivos JS

Cada módulo tem uma responsabilidade clara. Clique para explorar o que cada um faz.

🗄️
state.js
CORE · Estado Global

O "cérebro" de dados da aplicação. Armazena todas as variáveis compartilhadas entre módulos: TVs, categorias, usuário logado, arquivos selecionados, modo de upload e estado da playlist.

Exporta getters e setters para cada variável de estado
Gerencia: categories, tvs, currentUserId, selectedCategoryId, uploadMode, playlistEnabled, dzSelectedFiles
Qualquer módulo pode ler/escrever estado sem acoplamento direto
DEPENDÊNCIAS
nenhuma

firebase.js / auth2.js
FIREBASE · Inicialização

Inicializa o Firebase SDK (Auth, Realtime Database, Storage) e expõe globalmente via window.authModule. É o primeiro script a carregar no painel, criando a ponte com todos os serviços Firebase.

Cria a instância única do Firebase (guard !firebase.apps.length)
Expõe: auth, database, storage, onAuthStateChanged, signOut
Todos os módulos acessam Firebase via window.authModule

UI · Gerenciamento de TVs

O módulo mais completo. Gerencia todo o ciclo de vida das TVs: criar, deletar, ligar/desligar, editar chave de ativação, abrir modal de upload e visualizar mídia. Opera sobre o grid de TVs do painel.

Como o Sistema Funciona

Do login até a mídia aparecer na TV — passo a passo.

PASSO 01 · AUTENTICAÇÃO
Login via Firebase Auth

O usuário acessa index.html. O login.js escuta o submit do form, valida o email e chama signInWithEmailAndPassword(). O firebase.js já está carregado e disponibiliza window.authModule. Ao autenticar, o onAuthStateChanged redireciona para painel.html.

login.js → firebase.auth.signInWithEmailAndPassword() → redirect painel.html
PASSO 02 · INICIALIZAÇÃO
Painel carrega e inicializa módulos

O main.js espera o DOM e inicializa todos os módulos em ordem. O auth.js detecta o usuário logado, define o currentUserId (email com pontos trocados por vírgulas) e dispara a sincronização inicial com Firebase.

main.js → initAuth() → syncWithFirebase() → setCategories() + setTVs()
PASSO 03 · SINCRONIZAÇÃO
Dados chegam do Firebase

O firebase-sync.js busca users/{userId}/categories e users/{userId}/tvs, armazena no estado via state.js e chama ui-render.js para atualizar a sidebar e o grid. Para cada TV, calcula quais mídias estão ativas.

syncWithFirebase() → setCategories() + setTVs() → updateCategoryList() + updateTvGrid()
PASSO 04 · UPLOAD DE MÍDIA
Usuário envia arquivo para a TV

Clica em "Enviar mídia". O tv-manager.js abre o modal. O dropzone.js recebe o arquivo. O usuário clica em enviar: o upload-handler.js verifica a cota (via media-manager.js), faz upload para Storage, registra no DB e atualiza o nó midia/{activationKey} que o dispositivo físico escuta.

canUpload() → uploadMediaToStorage() → registerMediaInDB() → midia/{key}.set()
PASSO 05 · DISPOSITIVO RECEBE
TV exibe o conteúdo

O app Android/Windows instalado na TV escuta o nó midia/{activationKey} em tempo real. Quando detecta mudança, exibe a nova mídia (imagem, vídeo ou playlist). Toda a comunicação é unidirecional via Firebase Realtime Database.

Firebase Realtime DB → App DSigner (TV) → exibe mídia
// BANCO DE DADOS

Estrutura do Firebase

Como os dados estão organizados no Realtime Database.

dsignertv-default-rtdb/
users/ // dados de cada usuário (chave = email com '.' → ',')
{userId}/
categories/ { id, name, status } // grupos de TVs
tvs/ { id, name, categoryId, status, activationKey, media, playlist }
tv_midias/ // metadados de todas mídias enviadas
{tvSlug}/ {mediaName}/ { url, type, active, storagePath, timestamp... }
plan/ { label, quota } // plano de armazenamento
billing/ { status, planLabel, price, qrCodeUrl, pixKey... }
midia/ // canal de comunicação com os dispositivos físicos (TVs)
{activationKey}/ { tipo: 'image'|'video'|'playlist'|'stop', url, items, timestamp }
support_tickets/ // chamados de suporte
{ticketId}/ { userId, email, subject, message, status, messages/, unreadDev, unreadUser }
